Canonical Overview

Nori 是一个面向小红书内容生产的多 Agent 运行系统。

当前架构已经从“很多平铺函数和兼容文件”收敛成更清晰的层次：用户输入进入 session， 信息层编译 context，Agent 层负责画像、市场、规划、生成和复盘，workflow 层只管粗粒度执行、 Human Gate 和运行记录，LLM/OSS/storage 作为基础设施被注入使用。

ARCHITECTURE PRINCIPLE Domain logic stays inside agents. LangGraph 不应该拆成很多微 node；它只承载检查点、重试、Human Gate 和 artifact 记录。
RUNTIME PRINCIPLE Session owns continuity. 一个用户可以有多个 session；memory 保存长期事实，context 只装配本次调用需要的材料。
GENERATION PRINCIPLE Spec first, artifact second. 先冻结 ContentDesignSpec，再交给 Note、Cover、Package 等专门 agent 生成具体产物。
IMAGE PRINCIPLE Reference images must be fetchable. 本地参考图先上传 OSS，relay 图生图用 extra_body.reference_images。

系统分层

这张图是现在最重要的 mental model：不要再把 domain、context、memory、workflow、LLM gateway 混成一团。每层有明确职责，跨层只能通过 typed contract 或 adapter。

WHY

让业务意图先被建模

用户不是直接触发某个 prompt，而是形成 profile、task、assets、constraints 和 decision。这样每个 agent 接收到的是业务对象，而不是临时拼出来的字符串。

CONTEXT

Context 不是 Memory

Memory 是可持续沉淀的事实和偏好；Context 是某次 agent call 的切片视图。Context 可以引用 memory，但不应该替代 memory。

GRAPH

LangGraph 只做粗节点

Workflow node 应该对应阶段边界，比如 spec、generate、review、human gate。helper、normalizer、prompt builder 不应该被拆成 graph node。

QUALITY

Review Gate 是系统部件

生成结果进入 export/publish 前需要 review gate。测试默认跳过 Human Gate；产品模式可以 pause 并等待人工确认。

代码框架 SVG

这张图回答“看 repo 时先看哪里、每个包负责什么、允许怎样依赖”。它不是把所有文件列出来，而是把当前代码组织成稳定的 owner boundary，方便后续继续删兼容层、补测试和加产品入口。

01 / ENTRY scripts/, future UI/API 只负责把用户输入、数据目录和运行配置交给 session/workflow，不承载业务 prompt。
02 / HARNESS nori/workflows 粗粒度 stage、checkpoint、Human Gate、run record。LangGraph 是实现，不是业务边界。
03 / INFO nori/sessions + nori/context Session 保存连续工作状态；Context 只装配一次 agent call 需要的信息切片。
04 / AGENTS nori/agents/* 业务 prompt、policy、normalizer、package 和 review 逻辑跟各自 agent 放在一起。
05 / INFRA llms + nori/storage 封装 provider 和 OSS。Agent 使用稳定 facade，不直接知道 SDK 细节。

新增代码放哪里？

新的业务能力 优先放进对应 nori/agents/<domain>/<agent> package；prompt、input model、normalizer、policy 不要拆到顶层。
跨 agent 共享 contract 如果是稳定业务模型，放 nori/core/*_models.py；如果只是纯工具函数，放 nori/shared，并保持无状态。
一次运行的信息装配 放 nori/context。它可以引用 memory/session/project，但不要把长期事实写回 context。
跨 session 记忆 放 nori/memory。检索、promotion、serialization 都属于这一层，不要散到 agent helper。

依赖边界

Workflow -> Agents Workflow 只编排阶段和记录结果，不应该知道 NoteMaker/CoverDirector 内部 prompt 结构。
Agents -> Core/Context Agent 接收 typed context 和 typed model，内部完成 prompt、LLM call、fallback 和输出 normalize。
Agents -> LLM/Storage 通过 LLMFactory、image facade、ReferenceImagePublisher 等 facade 使用外部能力。
Removed legacy paths nori/capabilities、nori/core/models.py、nori/skill_base 和 planning facade 兼容入口不再作为新增代码入口。

业务流程

现在的内容生产不是一个“大生成 agent”，而是一条可检查的流水线。前面负责理解和规划，中间冻结 spec，后面才生成 copy、cover、package，最后进入 review 和 learning loop。

01 / INTAKE 理解用户与素材 Intaker 读取用户输入、图片、品牌素材和平台诉求，输出结构化 intake。 nori.agents.user_profiling.intaker
02 / PROFILE 形成账号画像 AccountPlanner 把用户、品牌、竞品和内容方向收敛成账号运营画像。 account_planner
03 / MARKET 学习平台内容技能 XHSNoteAnalyzer 从样本中抽取 NoteSkill、证据、聚类和可复用风格规则。 market_analysis
04 / PLAN 生成运营计划 Operation/KPI/Calendar Planner 输出任务、节奏、指标和内容日历。 planning
05 / SPEC 冻结生成规格 ContentSpecAgent 用 ContextView 生成 ContentDesignSpec，明确标题、结构、视觉和验收项。 spec_designer
06 / PACKAGE 生产并审查 ArtifactGenerationAgent 调用 NoteMaker、CoverDirector、ContentProducer，再进入 review gate。 content_generation + learning_loop

Session Runtime

上层架构是静态能力图，session 是用户的一次连续工作空间。一个用户可以有很多 session，每个 session 可以有多轮 turn、多条 task goal、多次 workflow run，并把关键结果提升到 memory。

Session 用户的一次工作空间。适合保存本轮对话、任务目标、阶段事件、运行状态和可恢复信息。
Turn 一次用户/系统交互。它不应该变成长期知识，只记录本轮输入输出和对 task 的影响。
TaskGoal session 内正在追踪的具体目标，比如生成某个小红书内容包、学习一批样本、复盘一次发布结果。
WorkflowRun 运行时执行记录，包含 stage status、Human Gate 状态、artifact refs。它不是业务 agent 本身。

Human Gate 应该放在哪里？

Human Gate 属于 workflow/runtime 层，不属于某个业务 agent。agent 可以产出需要确认的候选、风险或 checklist；workflow 决定是否暂停。

human_gate_mode="skip" 是默认测试模式；pause 才会记录 waiting_for_human 并等待产品界面恢复。

OSS Reference Image Path

图生图的关键不是“代码里有 reference_images 字段”，而是外部模型真的能读取参考图。现在本地素材会通过 storage 层发布到 Volcengine TOS，再以 signed HTTPS URL 传给 image provider。

Configured ReferenceImagePublisher.from_env() creates a TOS store from env and uploads local reference images before generation.
Not configured The publisher stays disabled and keeps old local-bytes behavior, so non-relay providers and offline tests still work.
Provider field Relay image generation sends URLs as extra_body.reference_images first. Older image_urls, images, and image are only fallbacks.
Verification Fidelity has to be checked visually with reference-only elements. Metadata such as reference_images_sent=true is useful but not sufficient proof.

为什么这层要独立？

CoverDirector 不应该知道 TOS SDK，也不应该保存 AK/SK。它只需要一个 publisher，把本地素材变成模型可读取的 reference input。

当前 storage 是 runtime asset publisher，不是完整 DAM。后面如果做素材库、权限、生命周期清理，可以继续扩展这一层。

代码目录地图

下面是当前最应该稳定下来的 owner boundary。以后新增能力时，先判断它属于哪一层，再决定放在哪里；不要为了一个 helper 重新开一层，也不要把业务规则塞进 infra。

CORE nori/core 基础 contract、AgentBase、WorkflowBase、LLMFactory、profile/asset/planning/capability models、project aggregate。
SESSION nori/sessions Session、Turn、TaskGoal、SessionEvent 和 SessionManager。负责用户连续工作空间。
CONTEXT nori/context ContextCompiler、ContextPackBuilder、ContextResolver、ContextBundle adapter。负责本次调用的信息装配。
MEMORY nori/memory StableProfile、SessionMemory、TaskMemory、retrieval、promotion、serialization。负责跨 session 沉淀。
WORKFLOW nori/workflows WorkflowSpec、StageSpec、HumanGateSpec、WorkflowRunner、WorkflowRun/StageRun 记录，以及 content_production 产品级编排。
USER PROFILING nori/agents/user_profiling Intaker 和 AccountPlanner。负责从用户输入、素材和品牌信息形成可运营画像。
MARKET nori/agents/market_analysis XHSNoteAnalyzer、NoteSkill、SessionSkillReport、样本聚类和平台内容技能学习。
PLANNING nori/agents/planning OperationPlanner、KPIPlanner、CalendarPlanner、policy、normalizer、task_builder。
GENERATION nori/agents/content_generation ContentSpecAgent、ArtifactGenerationAgent、NoteMaker、CoverDirector、ContentProducer 和 generation models。
LEARNING LOOP nori/agents/learning_loop Review gate、policy/scoring/state、strategy iteration 和 capability snapshot。
LLM GATEWAY llms chat/json/image facade、provider dispatch、capability guard、result normalization、telemetry。
STORAGE nori/storage ReferenceImagePublisher、Volcengine TOS adapter、object key convention。负责 runtime reference asset publishing。
Design note
这份 overview 是当前架构的对齐页：业务规则归 agents，运行时归 workflows，持续知识归 memory，本次输入归 context，外部服务归 infra。
