当前代码架构：运行时骨架 + Agent 能力组

这个文件名仍叫 expected，但内容已经更新为当前目标状态和已落地结构。 关键变化是：不再引入 Domains；`backend/` 是产品服务适配层；`memory/` 与 `context/` 是一级目录；`workflows/` 已接入 LangGraph；业务能力由 `nori.agents.*` 拥有。

四条架构原则

它们决定以后新增代码应该放在哪里，也决定哪些文件不该再出现。

P1

Runtime objects before scripts

先创建 session、task、context、workflow run，再让脚本或 API 调用 agent。脚本只是入口，不应该持有系统状态。

P2

Context is not memory

Memory 保存可复用事实；Context 只为当前 agent call 组装必要输入。两者保持一级目录是合理的。

P3

Agents own business behavior

业务能力留在五个 agent package 内，facade 对外，package/policy/normalizer/state 对内。

P4

Framework at orchestration boundary

LangGraph 负责粗粒度业务 graph、Human Gate 和执行轨迹，不侵入 NoteMaker、CoverDirector 等业务 agent 的内部判断。

为什么没有 Domains

如果再加 `domains/`，会和 `agents/*` 形成双重业务命名。现在 agent capability 已经是业务边界。

为什么保留 package.py

`package.py` 不是杂物桶，它应该聚合同一 stage 的输入恢复、prompt 构造、引用选择和装配类。

以后新增文件怎么判断

跨 agent 的放 core/runtime；当前调用输入放 context；长期事实放 memory；业务规则放 owning agent package。

脚本应该更薄

`scripts/run_holly_live_case.py` 现在只是 Holly live adapter；内容生产阶段顺序和 Human Gate 由 `nori/workflows/content_production` 拥有。

关键契约

这些类是系统可恢复、可审计、可测试的核心。后续产品化时，数据库/API schema 应围绕它们演进。

AgentBase

统一 agent 的 stage_name、LLM 注入和 JSON helper。

  • 业务 run 签名仍可各自保持领域语义。
  • 避免 agent 直接硬编码 llms 调用。

Session / TaskGoal

一次用户工作空间和本次任务目标分开。

  • 一个用户可以有多个 session。
  • 一个 session 可以有多个 task。

ContextBundle

当前 agent call 的上下文快照。

  • 包含 sources、memory、artifacts、payload。
  • 带 trace，方便排查输入来源。

ContextCompiler

把 profile、task、market、skills、assets 编译成带 slices 的 `ContextPack`。

  • canonical owner 是 `nori.context`。
  • planning 只保留兼容导出。

attach_context_pack

把规划产出的业务 `ContextPack` 挂入 runtime `ContextBundle`。

  • 保持 ContextPack 和 ContextBundle 分层清楚。
  • 自动追加 source、payload 和 trace。

WorkflowSpec

声明式列出 workflow 名称、粗粒度 stage handler 和可选 Human Gate。

  • 由 WorkflowRunner 直接编译执行 LangGraph。
  • 脚本/API 不再复制手写循环。

workflow_spec_from_base

把 core 的 `WorkflowBase` 转成 runtime `WorkflowSpec`。

  • 依赖方向是 workflows -> core。
  • core 不导入 LangGraph 或 runtime backend。

HumanGateSpec

workflow runtime 的人工确认控制点，不是业务 agent。

  • 默认 `human_gate_mode=skip` 方便测试。
  • `pause` 模式停在 gate 且不标记失败。

WorkflowRun

记录一次真实执行轨迹。

  • status、stages、artifact_refs。
  • 能记录 `StoredArtifact`、`_artifact_ref` 和 `_artifact_refs`。
