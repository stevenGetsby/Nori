# Nori P0 后端信息处理设计与推进

本文档用于推进 **P0 后端业务功能**。当前阶段不做前端页面设计，只设计和实现后端的信息处理能力、数据结构、Agent 分工、任务状态、产物存储与测试验收。

你可以直接引用编号推进：

```text
设计 P0-INTENT-02
实现 P0-TASK-03
检查 P0-XHS-04 当前代码覆盖情况
把 P0-REVIEW-03 拆成 prompt 和代码任务
更新 P0-API 的数据模型
```

## P0 后端目标

P0 只做一条后端闭环：

```text
用户原始输入 / 素材
→ 输入归一化
→ 意图解析
→ 缺失信息澄清数据
→ 生成 ContentBrief
→ 任务规划
→ 可选 Research
→ 小红书图文生成
→ 自动 Review / 修正
→ 生成 ContentPackage
→ 版本快照
→ 资产入库
→ 导出 / 发布草稿箱接口预留
```

P0 不设计真实前端。所有“交互”在后端先抽象为：

- API 请求与响应
- CLI/test runner 输入输出
- 任务事件流
- 结构化 JSON 状态
- 可落盘的中间产物

## 个人本地版 P0 执行规则

个人本地版先不做前端、账号体系、额度体系、真实发布和云端数据库。所有能力先通过 Python API、CLI/test runner、本地文件和测试产物验证。

执行节奏固定为：

```text
1. 选一个最小功能编号
2. 设计数据结构和边界
3. 实现代码
4. 跑对应测试
5. 更新本文档状态和测试记录
6. 再进入下一个功能
```

本地版默认决策：

| 决策项 | 本地版取法 |
| --- | --- |
| 用户体系 | 固定 `user_id=local` |
| 前端 | 暂不实现，用 CLI/test runner 代替 |
| 存储 | 先用本地 JSON/文件目录，必要时再加 SQLite |
| 发布 | 不真发布，先实现导出和草稿箱接口契约 |
| Research | 默认可跳过，先保证生成闭环 |
| 配图 | 先保证封面 + 文案，配图模型预留 |
| 任务执行 | 单任务串行，先不做并发队列 |
| 验收方式 | 每个功能必须有最小自动化测试或可复现脚本 |

## P0 完成定义

P0 后端完成时，必须满足：

- 可以通过统一请求创建小红书图文生成任务。
- 后端能解析用户输入，产出 `IntentDraft`、缺失项和 `ContentBrief`。
- 不依赖前端，也能通过 CLI/test runner 完成澄清、跳过、继续执行。
- 任务有可恢复的状态、步骤事件和错误记录。
- 生成链路产出标准 `ContentPackage`，包含标题、正文、标签、封面、prompt、素材使用说明。
- Review 产出标准 `ReviewReport`，低质量封面至少自动修正一次。
- 每次生成和编辑都保存 `VersionSnapshot`。
- 成品进入 `AssetRecord`，可通过后端接口导出。
- 所有中间产物都有 manifest，测试时能完整复盘。
