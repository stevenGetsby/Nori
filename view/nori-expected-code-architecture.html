Nori expected code architecture

预期代码架构：从业务域拆分到运行时内核

目标不是把所有 Agent 换目录，而是把“公共能力、运行时状态、上下文组装、三段 Agent 入口、领域实现”拆成清晰层级。 这样每个用户可以拥有多个 session，每个 session 又可以拥有多个 task 和 workflow run。

架构结论： 未来 Nori 应该从当前的业务域 package，进化为 runtime-first 结构： `core` 稳定，`memory/sessions/workflows` 管状态，`agents` 管入口，现有 domain modules 管具体能力实现。
core/shared contracts, base classes, LLM gateway, artifact primitives
runtime/memory, sessions, workflow runs, checkpoints, context resolution
agents/understanding, generation, evaluation as three public orchestration entrypoints
domains/profile, market, planning, creation, review implementations behind agent APIs
apps/CLI, scripts, API handlers, demos and product-facing entrypoints

四条重构原则

这些原则决定未来目录怎么拆，也决定哪些模块应该保留在原地，哪些模块应该提升成 runtime 层。

Runtime first

先建模 session、task goal、workflow run、artifact，再让 agent 接入。否则系统只能跑单次脚本，不能恢复和审计。

Agent as public facade

外部只接触 Understanding、Generation、Evaluation 三个入口；子 Agent 和 domain helper 留在内部。

Domain implementation stays modular

市场分析、账号规划、内容生成、审核策略仍是独立实现，不把所有逻辑塞进一个巨大 Agent。

Contracts before prompts

先定义 StableProfile、TaskGoal、ContextPack、Artifact 等结构，再设计 prompt 和 LLM 流程。

Foundation Layer nori/core + nori/shared

models.py跨层契约：UserAsset、ContentTask、IntentContract 等。
agent.pyAgentBase、AgentInputPreparer、AgentPromptBuilder。

Runtime Layer nori/memory + nori/sessions + nori/workflows

memory/StableProfile、SessionMemory、MemoryPatch、promotion policy。
sessions/Session、Turn、TaskGoal、SessionManager、current state。
workflows/WorkflowRun、StageRun、WorkflowRunner、resume and replay。

Context Layer nori/context

resolver.py从 profile、session、task、artifacts 挑当前需要的上下文。
packs.pyContextPack、PlatformContext、TopicContext、MetricContext。
sources.py平台策略、热点、历史指标和外部数据源适配。

Agent Facade Layer nori/agents

understanding/建立或更新 StableProfile、TaskGoal、SessionMemory。
generation/统一内容生成入口，按 note/article/image/package 路由。
evaluation/质量、合规、指标和 strategy iteration 的统一入口。

Domain Implementation Layer nori/domains or existing packages

profile当前 user_profiling：intaker、account planner。
market + planning当前 market_analysis 和 context_building。
creation + learning当前 content_generation 和 learning_loop。

Application Layer scripts + APIs + demos

api/未来产品 API：create session, send turn, run workflow。
scripts/smoke、live case、migration、debug 工具。
view/架构图、demo、内部对齐页面。

第一阶段不建议大搬家

先新增 memory/、sessions/、workflows/、context/，然后让现有 Agent 接入这些 runtime 模型。

现有 package.py 不是问题

每个 stage 内部的 package.py 仍然合理。它是领域实现内部的类式 contract，不需要再拆回小文件。

真正要新增的是运行时骨架

现在最缺的不是生成能力，而是 session、task、workflow run、artifact manifest、memory promotion 这些运行时实体。

apps 层要薄

API、CLI、脚本都应该调用 session/workflow/agent facade，不直接拼接 NoteMaker、CoverDirector 等内部类。

关键代码契约

这些模型应该先写清楚。它们会成为后续重构的稳定锚点，避免继续用散 dict 传递状态。

StableProfile

跨 session 复用的账号/品牌长期状态。

  • workspace_id, profile_id, version
  • brand_facts, audience, positioning
  • content_pillars, platform_strategy
  • guardrails, preferences, learned_skills

Session

一次任务推进的工作空间，不是 chat log。

  • session_id, workspace_id, user_id
  • entry_type, status, title, summary
  • current_task_goal_id
  • created_at, last_active_at, closed_at

TaskGoal

当前要完成的目标，可以吸收现有 IntentContract。

  • goal_id, session_id, status
  • deliverables, must_include, constraints
  • success_criteria, source_turn_ids
  • promotion_candidates

WorkflowRun

系统实际执行轨迹，支持重跑和 resume。

  • run_id, session_id, goal_id
  • workflow_type, status, current_stage
  • context_snapshot_id
  • artifact_manifest_path

StageRun

每个 Agent/stage 的输入、输出和错误边界。

  • stage_run_id, run_id, stage_name
  • agent_name, input_artifact_ids
  • output_artifact_ids, status
  • error, started_at, ended_at

Artifact

所有中间产物和最终产物的索引。

  • artifact_id, session_id, run_id
  • type, stage_name, version
  • path, data, metadata
  • input_refs, source_refs

未来调用关系

产品入口应该走 session runtime，再走三段 Agent facade，而不是直接从 API 调某个内部 stage。

01 API / CLI

receive turn

用户进入 onboarding 或 interaction，创建或恢复 session。

02 SessionManager

resolve state

拿到 StableProfile、SessionMemory、当前 TaskGoal。

03 ContextResolver

build context

组装平台策略、热点、指标、历史 artifact 的 ContextPack。

04 WorkflowRunner

execute agents

按 workflow graph 调 Understanding、Generation、Evaluation。

05 MemoryPromotion

learn safely

从审核、指标和用户确认中提炼稳定 profile 更新。

迁移路线

为了降低风险，推荐分两条线推进：一条补 runtime，一条逐步收敛 Agent facade。

Phase A：先补运行时

  1. 新增 nori/sessions/models.py 和 nori/workflows/models.py。
  2. 把现有 IntentContract 映射成 TaskGoal。
  3. 扩展 ArtifactStore 支持 run-scoped manifest。
  4. 写一个 ContentWorkflowRunner 包住现有 GenerationAgent。
  5. 用测试证明 create/resume/replay 的最小闭环。

Phase B：再收敛入口

  1. 新增 UnderstandingAgent，包住 Intake + AccountPlanner + TaskGoal 更新。
  2. 保留 GenerationAgent，但让它吃 TaskGoal + ContextPack。
  3. 新增 PerformanceAnalysisAgent，包住 ReviewGate + Metrics + StrategyIteration。
  4. 把脚本和 API 改成只调用 session/workflow facade。
  5. 最后评估是否需要把现有业务包迁移到 domains/。
AVOID

不要为了“看起来统一”马上把所有目录移动到新层级。先把运行时模型落地； 等 Session / Workflow / Artifact 的边界稳定后，再迁移业务包。否则会产生大量路径 churn，但产品能力没有真正变清楚。
