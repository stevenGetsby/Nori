先拆清楚：静态架构 vs 运行时 Session

公共架构是稳定能力地图；session 是每次任务推进时生成的状态容器。它们不应该混成一个“巨大上下文”。

Static System Architecture

这部分相对稳定，定义产品能力和代码边界。它回答：系统能做什么，能力归谁负责。

InputOnboarding / Interaction，统一入口，不暴露内部 agent。
Memory长期品牌画像、平台策略、账号偏好、历史学习。
Context平台策略、热点采集、指标监控，按需组装成 ContextPack。
AgentUnderstanding / Generation / Evaluation 三段主能力。
Core模型契约、ArtifactStore、LLMFactory、Workflow 基类。

为什么 Session 不等于 Chat Log

Chat log 只记录对话文本；Session 要记录目标、上下文快照、工作流状态、产物索引、审核结果和记忆提升候选。

为什么 TaskGoal 要独立出来

同一个 session 里用户可能连续做多个任务。TaskGoal 独立后，修改目标不会污染长期 profile，也能追踪每次交付的验收标准。

为什么 WorkflowRun 要独立出来

同一个 TaskGoal 可能重跑、失败、resume 或 A/B 两套生成。WorkflowRun 是系统实际执行轨迹，不是用户意图本身。

为什么 Artifact 要独立出来

产物需要版本、路径、来源 stage 和可审计输入。它是 debug、复盘、resume 和评估闭环的基础。

三种记忆，不要混写

最容易出错的是把用户一次会话里的临时偏好直接写进长期 profile。推荐采用 promotion gate。

Stable Memory

跨 session 复用，代表长期确定的账号资产和策略。

  • 品牌定位、受众、人设、长期平台策略
  • 内容支柱、禁忌、品牌词、视觉偏好
  • 已经验证过的高质量 NoteSkill
  • 历史策略迭代后的稳定结论

Session Memory

只在当前 session 内稳定，保留讨论上下文和过程判断。

  • 用户本轮临时要求和口头修正
  • 已经否定的方向和原因
  • 当前上下文快照和中间选择
  • 待确认的记忆提升候选

Task Memory

围绕一个具体交付物，通常随任务完成而归档。

  • 本条内容的目标、交付物、平台和格式
  • must_include、taboos、success criteria
  • 本次 ContentTask / NoteDraft / CoverResult
  • 对应审核结果和指标快照
Session Observations用户说了什么、系统观察到什么、哪些需求只是本次有效。
→
Promotion Gate是否重复出现、是否被用户确认、是否与已有 profile 冲突。
→
StableProfile Patch形成可审计的 profile 版本更新，而不是直接覆盖旧记忆。

一次 Session 的生命周期

这个序列可以直接映射到未来代码里的 SessionManager、ContextResolver、WorkflowRunner 和 MemoryPromotion。

Overview Architecture Split Session Model Memory Types Lifecycle Code Plan
01

Create or Resume Session

根据 workspace 和 entry_type 创建 session；如果有未完成 run，则恢复 current stage。

Session.status = active
02

Resolve Memory Snapshot

读取 StableProfile、session notes 和最近 artifact，形成本轮可用上下文。

MemorySnapshot
03

Build or Update TaskGoal

把用户这次需求转成明确交付物、约束和验收条件；对应现有 IntentContract。

TaskGoal / IntentContract
04

Assemble ContextPack

按任务读取平台策略、热点、指标、历史内容和当前 profile，不把所有历史粗暴塞给 Agent。

ContextPack
05

Run Workflow

Understanding / Generation / Evaluation 按需执行；每个 stage 写 checkpoint 和 artifact。

WorkflowRun + StageRun
06

Review and Collect Metrics

质量审核、合规审核和人工指标都成为 session 内可追踪证据。

Review + MetricsSnapshot
07

Close Session or Continue

用户可能继续追问、修改目标、重跑生成；session 直到明确完成才归档。

paused / completed
08

Promote Memory

把被确认、重复出现或被指标验证的结论提升到 StableProfile 新版本。

StableProfile.version += 1

推荐落地目录

不建议立刻大搬家。先补 session runtime，再逐步把现有 IntentContract、ArtifactStore 和 workflow 接进去。

Memory Layer

nori/memory/
  • models.py: StableProfile, SessionMemory, MemoryPatch
  • store.py: profile/session memory persistence
  • promotion.py: session -> stable promotion policy

Session Runtime

nori/sessions/
  • models.py: Session, Turn, TaskGoal
  • manager.py: create/resume/close session
  • context.py: build ContextPack for current task

Workflow Runtime

nori/workflows/
  • models.py: WorkflowRun, StageRun
  • runner.py: execute/resume stage graph
  • content_loop.py: session-level content workflow

Artifact Runtime

nori/core/artifacts.py
  • extend ArtifactStore into run-scoped manifests
  • track input_refs and output_refs per StageRun
  • support replay/debug from artifact index
IMPLEMENTATION ORDER

第一阶段先只建模，不迁移所有 agent：新增 Session / TaskGoal / WorkflowRun / Artifact models， 把现有 IntentContract 映射成 TaskGoal，把 ArtifactStore 接成 WorkflowRun checkpoint。 第二阶段再把 UnderstandingAgent、GenerationAgent、PerformanceAnalysisAgent 作为三个统一入口收敛。
