Nori session runtime

每个 Session 是一个可恢复的任务工作空间

静态系统架构定义 Nori 有哪些能力；Session Runtime 定义某个用户这一次任务如何推进、执行到哪一步、产物在哪里、哪些事实可以沉淀为长期记忆。

当前代码已经有 `Session`、`TaskGoal`、`ContextBundle`、`WorkflowRun`、`MemoryRecord` 和 `RuntimeRunRecorder`； `ContextCompiler` 先生成业务 `ContextPack` / `ContextView`，`attach_context_pack` 再挂入 runtime，stage 产物通过 `StoredArtifact` 进入 workflow trace。 下一步产品化时，需要把这些 in-process 模型接到持久化存储和 API/UI。
USER
一个用户可以有多个 session，不同 session 代表不同任务工作空间。
SESSION
一次工作推进保存 turns、events、task_goals 和运行 metadata。
TASK
一个明确目标保存 goal、workflow_name、acceptance。
RUN
一次实际执行LangGraph workflow run，可成功、失败、等待人工或重跑。
ARTIFACT
可审计产物每个 stage 的输入输出快照和最终封面。

静态架构 vs Session Runtime

这两个层次必须分开。静态架构回答“系统能做什么”；Session Runtime 回答“这次任务做到哪一步”。

Static System Architecture

稳定、共享、低频变化。它定义代码边界和能力所有权。

CORE跨阶段契约、ContextPack、LLMFactory、AgentBase、ArtifactStore。nori/core
MEMORY长期事实、session/task 记忆、promotion policy。nori/memory
CONTEXT业务上下文编译、agent 视图投影、当前调用上下文组装。nori/context
AGENTS五个业务 capability 拥有具体 agent 行为。nori/agents
GRAPH跨阶段编排与执行记录。nori/workflows

Runtime Session Architecture

动态、用户相关、高频变化。它保存一次任务推进的过程状态和证据。

01Session 保存 user_id、profile_id、turns、events、metadata。active
02TaskGoal 保存本次目标、workflow_name、acceptance。goal
03ContextPack/View 保存业务上下文，ContextBundle 保存当前调用 envelope。context
04WorkflowRun 保存粗粒度节点、Human Gate、StoredArtifact refs、metadata、status。run
05MemoryRecord 经过 promotion 才进入 stable profile。learn

Session 数据模型

现在最合理的建模是：User 拥有 StableProfile 和多个 Session；每个 Session 可以包含多个 TaskGoal；每个 TaskGoal 可以产生多个 WorkflowRun。

User

  • user_id
  • workspace_id
  • 入口身份
产品层实体，代码里目前由调用方传入。

StableProfile

  • profile_id
  • brand_profile
  • preferences
  • constraints
跨 session 复用的长期状态。

Session

  • session_id
  • turns
  • task_goals
  • events
  • metadata
一次任务工作空间，不等于 chat log。

TaskGoal

  • task_id
  • goal
  • workflow_name
  • acceptance
  • status
本次交付物和验收条件。

Context

  • ContextPack
  • ContextView
  • ContextBundle
  • sources
  • trace
业务上下文和 runtime envelope 分开，必要时显式桥接。

WorkflowRun

  • run_id
  • status
  • stages
  • artifact_refs
  • metadata.human_gate_mode
LangGraph 实际执行轨迹，Human Gate 默认 skip。

一个 Session 可以有多个 Task

用户可能先做账号定位，再做一篇内容，再要求重写封面。TaskGoal 独立后，不会把所有需求混成一团。

一个 Task 可以有多个 Run

同一目标可以失败后恢复、切换参数重跑、或 A/B 两套生成。WorkflowRun 记录实际发生过什么。

Artifact 不是附属品

content_package、cover、review、summary 都是排查和产品化体验的证据，需要从 workflow 中被引用。

三种 Memory

Memory 和 Context 分成一级目录是正确的：Memory 管长期/中期事实，Context 管本次调用如何取用这些事实。

StableProfile

跨 session 稳定复用，代表账号、品牌、偏好和约束。

  • brand_profile
  • account_profile
  • preferences
  • facts

SessionMemory

只在当前 session 内稳定，保存过程判断和临时观察。

  • session_id
  • records
  • artifact_refs
  • metadata

TaskMemory

绑定某个 task goal，保存本次目标下的产物、约束和决策。

  • task_id
  • goal
  • records
  • artifact_refs
Session / Task Observations用户表达、系统观察、审核结果、指标反馈。
→
Promotion Gate被用户确认、重复出现、指标验证、无冲突。
→
StableProfile Patch形成可审计的长期 profile 更新，而不是直接覆盖。

一次 Session 生命周期

这条生命周期现在已经能映射到 `RuntimeRunRecorder`、`SessionManager`、`ContextResolver` 和 `WorkflowRunner`。

Static vs Runtime Data Model Memory Types Holly Snapshots
01

Create Session

`RuntimeRunRecorder.start()` 调用 `SessionManager.create_session()`，保存 user/profile/run_dir。

Session.status = active
02

Append Turn

把当前用户 brief 作为 turn 写入 session，保留来源 metadata。

Turn(role=user)
03

Start TaskGoal

创建 goal、workflow_name、acceptance，明确这次要交付什么。

TaskGoal
04

Compile / Resolve Context

`ContextCompiler.build()` 生成 ContextPack；`ContextResolver.for_agent()` 投影 ContextView；`ContextResolver.build()` 组装 runtime ContextBundle。

ContextPack -> ContextView -> ContextBundle
05

Run LangGraph Workflow

`WorkflowRunner().run(spec, state)` 执行粗粒度业务节点；节点内部保留细粒度 artifact trace。

WorkflowRun.stages
06

Optional Human Gate

默认 `human_gate_mode=skip` 直接记录 skipped gate；产品模式 `pause` 停下来等待人工确认。

HumanGateSpec
07

Record Artifacts

stage 返回 `StoredArtifact`、`_artifact_ref` 或 `_artifact_refs`，写入 StageRun.output_ref 和 WorkflowRun.artifact_refs。

artifact_refs[]
08

Write Snapshots

写出 `session.json`、`context_bundle.json`、`workflow_run.json`，失败时也保留已执行轨迹。

RuntimeRunRecorder.write_snapshot
09

Promote Memory Later

审核、指标或用户确认后的事实，经过 promotion policy 后进入 StableProfile。

MemoryPromotion
